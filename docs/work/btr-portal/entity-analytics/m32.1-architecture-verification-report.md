# M32.1 — Entity Analytics Architecture Verification Report

**Date:** 2026-06-24  
**Scope:** Pre-M32.2 architectural verification and targeted refactoring  
**References:** [entity-analytics-architecture.md](./entity-analytics-architecture.md), [implementation-plan-entity-analytics.md](./implementation-plan-entity-analytics.md), [m32.1-implementation-summary.md](./m32.1-implementation-summary.md)  
**Note:** Entity-pack milestone numbers below reflect the original plan. **Current sequence:** [entity-analytics-roadmap-authoritative.md](./entity-analytics-roadmap-authoritative.md) (Salesman = M32.9, Supplier = M32.10, Item = M32.11).

---

## Executive Summary

M32.1 delivered a sound plugin-oriented **producer orchestrator** and a **metadata-capable KPI registry**, but entity-type discovery was still hardcoded in platform services. Targeted refactoring introduced an **entity type registry** and **registrar bootstrap pattern** so future entity types extend the platform through registration — not conditional branching.

**Final verdict: READY FOR M32.2** (with documented minor limitations below).

---

## 1. KPI Registry Assessment

### Verdict: **PASS** (after refactoring)

### Architecture observations (before refactoring)

| Check | Initial state |
| ----- | ------------- |
| KPI definitions independent of entity types | **PASS** — `EntityKpiMetadata` stored by `KpiId` only |
| KPI packs reference KPI IDs | **PASS** — `RegisterPack(packId, kpiIds)` |
| New packs without registry code changes | **PASS** — dictionary-backed `RegisterPack` |
| Entity type → pack linkage | **FAIL** — no linkage; `EntityTypeRegistration` model unused |
| Entity type registration metadata-driven | **FAIL** — `EntityTypeCode.KnownTypes` whitelist + hardcoded list in `EntityAnalyticsService` |
| Customer-specific registry logic | **PASS** — none in registry |

The KPI registry core (`EntityAnalyticsKpiRegistry`) was already metadata-driven with no `if (entityType == Customer)` branching. The gap was **entity type → KPI pack resolution** and **entity type discovery** living outside the registry model.

### Design weaknesses addressed

1. **`EntityTypeCode.TryParse`** enforced a fixed four-type whitelist — blocked Warehouse/Wilayah/etc. without code edits.
2. **`EntityAnalyticsService.GetEnabledTypes()`** hardcoded `Customer | Salesman | Item | Supplier`.
3. **`EntityTypeRegistration`** existed but was never registered or consumed.

### Refactoring performed

| Change | Purpose |
| ------ | ------- |
| Added `IEntityTypeRegistry` + `EntityTypeRegistry` | Metadata-driven entity type catalog |
| Added `IEntityAnalyticsRegistrar` + `EntityAnalyticsPlatformRegistrar` | Open/closed registration hook |
| Added `EntityAnalyticsRegistryBootstrap` | Runs all registrars at startup |
| Extended `IKpiRegistry` with `GetPackKpiIdsForEntityType`, `ResolvePackMetadata` | Entity Type → KPI Pack → KPI IDs chain |
| Injected `IEntityTypeRegistry` into `EntityAnalyticsKpiRegistry` | Pack resolution without entity branching |
| Refactored `EntityPerformanceProfileComposer` and `EntityAnalyticsService` | Use registry instead of `EntityTypeCode.TryParse` / hardcoded arrays |
| Simplified `EntityTypeCode` | Constants only (including future type names); no validation whitelist |

### Resulting model

```text
EntityTypeRegistration (IEntityTypeRegistry)
        ↓ KpiPackId
EntityAnalyticsKpiRegistry.RegisterPack
        ↓ List<KpiId>
EntityKpiMetadata (by KpiId)
```

No customer KPI definitions or pack contents were added (per constraints).

## 2. Producer Architecture Assessment

### Verdict: **PASS**

### Architecture observations

| Check | State |
| ----- | ----- |
| Producer discovery via DI | **PASS** — `IEnumerable<IEntityAnalyticsProducer>` injected into orchestrator |
| No hardcoded entity routing | **PASS** — no `switch` / `if (domain == Customer)` in orchestrator |
| New producer = implement + register | **PASS** (improved) — Scrutor auto-registers `IEntityAnalyticsProducer` implementations |
| Orchestrator unchanged for new types | **PASS** |

`EntityAnalyticsProducerOrchestrator` filters producers by `WorkerDomain` or `EntityType` properties declared on each implementation — discovery is data-driven from registered plugins.

### Refactoring performed

| Change | Purpose |
| ------ | ------- |
| Scrutor scan for `IEntityAnalyticsProducer` | M32.2 adds producer class only — no orchestrator/DI list edits required |
| Added `ProduceForEntityType` | Optional entity-type-keyed invocation without domain coupling |
| Injected `IEntityTypeRegistry` into orchestrator | Normalize entity type codes before producer matching |

No producer implementations were added (per constraints).

## 3. Extensibility Assessment

### Adding a new entity type (e.g. Warehouse — M32.6)

| Layer | Changes required | Open/Closed |
| ----- | ---------------- | ----------- |
| **Platform (composer, API, orchestrator)** | None | ✓ |
| **Entity type registry** | Add `IEntityAnalyticsRegistrar` implementation OR extend platform registrar | ✓ (registrar only) |
| **KPI registry** | Registrar calls `RegisterMetadata` + `RegisterPack` | ✓ |
| **Producer** | Implement `IEntityAnalyticsProducer` (auto-discovered) | ✓ |
| **Worker** | Hook orchestrator in domain worker (one line per worker domain) | Expected — worker integration is external |
| **API** | None — generic `/{entityType}/{entityId}` | ✓ |
| **UI** | Optional entity-specific view route; generic shell works today | Minor — route template from registration |

### Per planned entity type

| Entity | Producer | KPI Pack registrar | DI registration | Platform code edit |
| ------ | -------- | ------------------ | ----------------- | ------------------ |
| Customer (M32.2) | `CustomerEntityAnalyticsProducer` | `CustomerEntityAnalyticsRegistrar` | Auto (producer) + registrar `AddSingleton` | None |
| Salesman (M32.3) | `SalesmanEntityAnalyticsProducer` | `SalesmanEntityAnalyticsRegistrar` | Same pattern | None |
| Item (M32.5) | `ItemEntityAnalyticsProducer` | `ItemEntityAnalyticsRegistrar` | Same pattern | None |
| Supplier (M32.4) | `SupplierEntityAnalyticsProducer` | `SupplierEntityAnalyticsRegistrar` | Same pattern | None |
| Warehouse, Wilayah, Brand, Category, Collector | Per-entity producer | Per-entity registrar | Auto + registrar | None |

### Open/Closed violations remaining (minor limitations)

| Limitation | Impact | Mitigation in M32.2+ |
| ---------- | ------ | -------------------- |
| `EntityAnalyticsPlatformRegistrar` lists initial four types | New types need a registrar class | Add `WarehouseEntityAnalyticsRegistrar` — no platform service edits |
| M32.2 registrar must be registered in DI | One `AddSingleton<IEntityAnalyticsRegistrar, …>` line | Acceptable; could move to Scrutor scan for registrars later |
| Worker hook not yet wired | Expected — M32.1 scope | Single orchestrator call in Customer worker |
| Frontend home view is generic | No per-type UX yet | M32.2 customer profile view |
| `EnabledEntityTypes` in appsettings | Feature flag per type | Configuration, not code branching |

## 5. Final Verdict

### **READY FOR M32.2**

### Justification

1. **KPI registry** now supports the intended `Entity Type → KPI Pack → KPI Definitions` chain without entity-specific branching.
2. **Producer architecture** follows the plugin model with DI discovery; orchestrator requires no modification per entity.
3. **Platform services** (composer, types API) read from registries — not hardcoded entity lists.
4. **M32.2 work** can proceed as: `CustomerEntityAnalyticsRegistrar` + `CustomerEntityAnalyticsProducer` + worker hook + KPI pack content — without refactoring platform infrastructure again.

### Minor limitations (acceptable for M32.2)

- Each new entity milestone adds an `IEntityAnalyticsRegistrar` with one DI line (or future Scrutor scan).
- Worker domain workers need a one-line orchestrator hook (documented in M32.1 summary).
- KPI pack catalog ID validation CI test deferred until M32.2 pack content exists.

## Recommended M32.2 Registration Pattern

```csharp
// CustomerEntityAnalyticsRegistrar.cs
public class CustomerEntityAnalyticsRegistrar : IEntityAnalyticsRegistrar
{
    public void Register(IEntityTypeRegistry entityTypes, IKpiRegistry kpiRegistry)
    {
        // KPI metadata + customer-default pack IDs only — no platform edits
        kpiRegistry.RegisterMetadata(...);
        kpiRegistry.RegisterPack("customer-default", new[] { "CU-KPI-009", ... });
    }
}

// CustomerEntityAnalyticsProducer.cs — auto-registered via Scrutor
public class CustomerEntityAnalyticsProducer : IEntityAnalyticsProducer { ... }
```

```csharp
// ApplicationPortalExtensions.cs (M32.2)
services.AddSingleton<IEntityAnalyticsRegistrar, CustomerEntityAnalyticsRegistrar>();
```

No changes to `EntityPerformanceProfileComposer`, `EntityAnalyticsProducerOrchestrator`, or `EntityAnalyticsController` required.
