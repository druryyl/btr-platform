# ADR-004: Idempotent Snapshot Replacement

**Status:** Accepted  
**Date:** 2026-06-25  
**Milestone:** M32.B1

---

## Context

Production backfill must be safe to **rerun** after failure, validation rejection, or formula fixes. Executing January 2025 twice must yield identical snapshot rows.

The live Entity Analytics persistence layer uses these semantics:

- `SaveMonthlyHistory` — SQL MERGE that **updates only when `IsClosed = 0`**
- `CloseMonth` — sets `IsClosed = 1` on all L1 rows for a period
- `ReplaceCurrentMetrics` — delete-all + insert for L0 CURRENT

Historical backfill writes months that are **born closed** (`IsClosed = 1`). A naive rerun using live `SaveMonthlyHistory` would **silently skip** updates on closed rows — violating idempotency.

---

## Decision

### 1. Backfill uses period-scoped replace, not live upsert

Introduce backfill-specific repository methods:

| Method | Behavior |
| ------ | -------- |
| `ReplaceMonthlyHistoryForPeriod(entityType, year, month, rows)` | DELETE L1 rows for period → INSERT new rows with `IsClosed = 1` |
| `ReplaceRankingForPeriod(entityType, year, month, rows)` | DELETE L2 for period → INSERT |
| `ReplaceAttentionForPeriod(...)` | DELETE L3 period slice → INSERT |
| `ReplaceRelationshipForPeriod(...)` | DELETE L4 period slice → INSERT |
| `ReplaceRadarForPeriod(...)` | DELETE L5 for period → INSERT |

Invoked only when `EntityAnalyticsProduceContext.Replay != null`.

### 2. Live paths unchanged

When `Replay == null`, producers continue using `SaveMonthlyHistory`, `CloseMonth`, and existing engine persist methods.

### 3. Force rerun

CLI `--force` triggers replace even when checkpoint status is `Completed`.

### 4. Deterministic computation

Within a period batch:

- Entity population sorted by `EntityCode` before ranking
- No `DateTime.Now` inside aggregator formulas
- `GeneratedAt` may differ between runs without affecting KPI `NumericValue` / `TextValue` equality

### 5. Transaction boundary

One database transaction per `(EntityType, Period)` for L1 + L2 + L5 writes — rollback on failure leaves no partial period.

---

## Consequences

### Positive

- True idempotency for closed historical months.
- Safe recovery after bugs or data fixes via `--force --from-period`.
- Clear separation between live open-month upsert and backfill replace.

### Negative

- Additional repository methods to implement and test.
- DELETE + INSERT generates more log activity than in-place MERGE — acceptable for one-time job.

---

## Alternatives Considered

| Alternative | Rejected because |
| ----------- | ---------------- |
| Temporarily set `IsClosed = 0` before live MERGE | Race with live worker; hacky |
| Live MERGE only | Cannot update closed rows — fails idempotency |
| Separate backfill tables | Duplicates read path; violates reuse objective |
| Version column per row | Over-engineered for one-time job |

---

## Compliance

Addresses architecture tasks **§7 Idempotency** and **§6 Checkpoint Strategy** (safe rerun).
