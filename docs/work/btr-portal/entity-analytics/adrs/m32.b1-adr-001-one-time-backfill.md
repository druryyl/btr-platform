# ADR-001: One-Time Backfill Worker Separation

**Status:** Accepted  
**Date:** 2026-06-25  
**Milestone:** M32.B1  
**Deciders:** Solution Architect, Product Owner (constraint)

---

## Context

Entity Analytics requires 36 months of historical snapshots at initial deployment. Management has confirmed this is a **one-time seeding** operation, not an ongoing operational capability.

The live platform uses scheduled domain workers (`RefreshDashboardCustomerSnapshotWorker`, etc.) that:

- Run on current business date
- Write L0 CURRENT and upsert open-month L1
- Invoke `EntityAnalyticsMonthCloseService` for prior-month close and retention purge

Historical replay must not alter this runtime architecture or be accidentally scheduled alongside normal refreshes.

---

## Decision

1. Introduce a **dedicated** `EntityAnalyticsHistoricalBackfillWorker` registered as CLI domain `EntityAnalyticsHistoricalBackfill`.
2. The backfill worker is **never** included in `--domain All` or Task Scheduler jobs.
3. Backfill is triggered **only** by manual administrator execution with explicit `--confirm BACKFILL`.
4. Backfill writes **L1–L5 for closed historical months only** — it does not replace the live worker's responsibility for L0 CURRENT or open-month accumulation.
5. After backfill completes, **only** existing scheduled workers continue Entity Analytics forward.

---

## Consequences

### Positive

- Clear operational boundary — no risk of recurring full-history replay load.
- Live worker code paths remain unchanged when `Replay == null`.
- Audit trail via dedicated `BackfillJob` records distinct from daily refresh logs.

### Negative

- Operators must remember backfill is a separate command — mitigated by runbook.
- Two worker entry points to document (live domain workers vs backfill domain).

### Neutral

- No new profile API or read-path changes.

---

## Alternatives Considered

| Alternative | Rejected because |
| ----------- | ---------------- |
| Flag on existing Customer worker `--backfill` | Couples one-time migration to daily worker; risk of accidental scheduling |
| Scheduled monthly backfill job | Violates one-time requirement; unnecessary after seed |
| SQL script-only migration | Bypasses producers/engines; ranking/radar would not match live semantics |

---

## Compliance

Satisfies approved constraint **§1 One-Time Backfill**.
