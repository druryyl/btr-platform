# ADR-005: Checkpoint Resume Strategy

**Status:** Accepted  
**Date:** 2026-06-25  
**Milestone:** M32.B1

---

## Context

Historical backfill for 36 months across four entity types may run **24–72 hours** total (Item-dominated). Process crashes, database timeouts, operator cancellation, and maintenance windows are expected.

The job must resume without:

- Reprocessing completed months (wasted time and DB load)
- Leaving partial corrupt periods (broken L2 ranks)
- Duplicating rows on resume

---

## Decision

### 1. Checkpoint grain: `(EntityType, PeriodYear, PeriodMonth)`

The smallest resumable unit is **one entity type for one closed month** — all entities in population processed together.

Sub-entity checkpointing (per CustomerId) is **not** supported in initial implementation.

### 2. Checkpoint states

| Status | Meaning |
| ------ | ------- |
| `Pending` | Not started |
| `Running` | In progress (set at period start) |
| `Completed` | L1 + requested layers persisted and month closed |
| `Failed` | Period transaction rolled back; `LastError` populated |
| `Skipped` | Dry-run or operator skip |
| `Cancelled` | Cooperative stop mid-job |

On resume, `Running` is treated as `Failed` (previous crash) and re-executed.

### 3. Resume default: skip completed

`--resume true` (default) skips periods with `Completed` status. `--force` reprocesses regardless.

### 4. Restart vs resume

| Flag | Behavior |
| ---- | -------- |
| `--resume` | Continue job from last incomplete period |
| `--restart` | Clear checkpoints for selected scope; snapshot data untouched unless `--force` |

### 5. Failure policy

Default: **stop on first period failure**. Optional `--continue-on-error` logs failure and proceeds — for non-production only.

### 6. Persistence

Dedicated tables:

- `BTRPD_EntityAnalytics_BackfillJob` — job-level metadata
- `BTRPD_EntityAnalytics_BackfillCheckpoint` — per-period status

Linked to `BTRPD_RefreshLog` via `LastRefreshLogId` for duration and error diagnostics.

### 7. Idempotent resume

Period re-execution uses ADR-004 replace semantics — safe whether previous attempt committed partially (should not occur due to transaction boundary) or failed before commit.

---

## Consequences

### Positive

- Operators can stop and continue across nights/weekends.
- Clear visibility into progress (e.g., "Customer ✓ through 2024-05").
- Failed period pinpointed for targeted fix and `--force` rerun.

### Negative

- Failed period must be reprocessed entirely — even if failure was one bad entity (fail-fast design).
- Checkpoint tables are migration artifacts remaining after backfill — low storage cost; useful for audit.

---

## Alternatives Considered

| Alternative | Rejected because |
| ----------- | ---------------- |
| Per-entity checkpoint | Explodes row count; incompatible with L2 population requirement |
| File-based checkpoint | Not auditable; not visible to ops tooling |
| No checkpoint — manual `--from-period` | Error-prone for 36×4 iterations |
| Continue on error (default) | Risk of silent gaps in history |
