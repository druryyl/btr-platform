# ADR-003: Replay Ordering (Month-by-Month, Oldest First)

**Status:** Accepted  
**Date:** 2026-06-25  
**Milestone:** M32.B1

---

## Context

Historical backfill must populate L1 (monthly KPIs), L2 (ranking history), and L5 (radar) for up to 36 months × four entity types.

Several iteration strategies are possible:

- Oldest → newest vs newest → oldest
- Month-by-month (full population) vs entity-by-entity (single entity across all months)
- Salesman RepHistory fast path vs full transaction replay

---

## Decision

### Primary loop: month-by-month, oldest → newest

For each entity type, iterate closed calendar months from `(Today − RetentionMonths)` through `LastClosedMonth` in **ascending** order. For each month:

1. Load all transactional data for the full entity population.
2. Run domain aggregator once.
3. Run producer + L2/L5 engines for the full population.
4. Close month and checkpoint.

### Entity type order: Salesman → Customer → Supplier → Item

Salesman first to validate infrastructure via RepHistory fast path. Item last due to cardinality and runtime.

### Ranking and radar require full population per period

L2 (`EntityRankingEngine`) and L5 (`EntityRadarEngine`) compute peer-relative positions across **all eligible entities** for a period. Entity-by-entity iteration would produce incorrect or incomplete ranks if used as the primary loop.

### Salesman L1 hybrid

When `BTRPD_SalesmanRepHistory` contains a row for `(SalesPersonId, PeriodYear, PeriodMonth)`, migrate directly to `BTRPD_EntityAnalytics_Monthly` without transaction replay. Use transaction replay only for gaps before RepHistory coverage.

### Parallelism

Default `max-parallel = 1` (sequential periods). Parallel month execution is **not** in initial scope.

---

## Consequences

### Positive

- Chronological processing matches month-close semantics and trend validation.
- One aggregation pass per month minimizes DAL load vs entity-by-entity.
- Correct L2/L5 peer distributions per period.
- Salesman fast path reduces total runtime by hours.

### Negative

- Failure mid-entity-type leaves later months pending — mitigated by checkpoints.
- Cannot "finish one customer completely" in a single pass — acceptable; not a business requirement.

---

## Alternatives Considered

| Alternative | Rejected because |
| ----------- | ---------------- |
| Newest → oldest | Harder to validate; month-close ordering unnatural |
| Entity-by-entity primary loop | Breaks L2/L5 population semantics; N× redundant DAL scans |
| Full transaction replay for all Salesman months | Ignores free RepHistory data; wastes hours |
| Parallel months | Risk of DB contention; ranking does not benefit |
