# M32.2R — Customer Performance Profile Hardening Summary

**Milestone:** M32.2R (refinement, not feature)  
**Date:** 2026-06-24  
**Status:** Complete  
**Prerequisite:** M32.2 Customer Performance Profile

---

## Objective

Stabilize the Entity Analytics platform as the baseline for M32.3–M32.8 without adding business capabilities or new KPIs.

---

## Hardening Summary

| Part | Outcome |
|------|---------|
| API contract | Additive fields only (`SnapshotVersion`, `ContractVersion`, KPI envelope metadata). Canonical `UnavailableReason` constants. 400 for invalid input/unknown type. |
| Snapshot metadata | `DefinitionVersion` populated on L0 rows. `HasAnyCurrentMetrics` excludes meta-only rows. Empty produce clears stale L0. |
| KPI metadata | Full metadata on customer pack: `ValueType`, `AggregationType`, `TimeGrain`, `DisplayPrecision`, `NullableBehavior`, `EvidenceFilterDimension`, `SourceDomain`, `ApplicableEntityTypes`. `ValidatePack()` for CI. |
| Entity identity | Profile `EntityId` resolves to `EntityCode` (CustomerCode-first). Repository identity uses business code consistently. |
| Repository | Layer-specific interfaces (L0–L5) with stub implementations. Dimension labels moved to `IDimensionLabelRegistry`. |
| Extension points | Registrar accepts dimension registry. `EntityKpiEnvelopeFormatter` extracted for reuse. Open/Closed preserved. |
| Code quality | Composer de-duplicated. Worker atomicity: entity analytics inside domain transaction scope. |
| Tests | +`EntityAnalyticsServiceTest`, pack validation, empty portfolio clear, shared repository stub base. |
| Documentation | [entity-analytics-developer-guide.md](../../features/entity-analytics/entity-analytics-developer-guide.md) |

---

## Architecture Observations

1. **Platform-first design holds** — Customer additions required only registrar/producer/resolver; composer and API unchanged in structure.
2. **L1–L5 seams are ready** — Row models and repository interfaces exist; SQL and engines deferred to M32.3+.
3. **Identity model is stable** — `(EntityType, EntityCode)` is the business key across API, snapshot, and routing.
4. **KPI envelope is the presentation contract** — Registry metadata projects to API; formatting centralized in `EntityKpiEnvelopeFormatter`.
5. **Enabled-type gate** — Config `EnabledEntityTypes` now enforced on profile read; types API already exposed `IsEnabled` / `IsAvailable`.

## API Contract Changes

**No breaking changes.** Additive only:

| Field | Location | Value |
|-------|----------|-------|
| `SnapshotVersion` | Profile root | `L0-v1` |
| `ContractVersion` | Profile root | `M32.2-v1` |
| `FilterDimension` | `KpiEnvelopeDto` | e.g. `customerCode` |
| `ValueType`, `DisplayPrecision`, `TrendEligible`, `RankEligible`, `NullableBehavior` | `KpiEnvelopeDto` | From registry |

**Behavioral (non-breaking):**

- Disabled entity types return `IsAvailable: false` with `EntityTypeDisabled` on all sections (was previously composable with partial data).
- Unknown entity type returns HTTP 400 (was unhandled 500).

---

## Repository Improvements

- `IEntityAnalyticsRepository` now composes six layer interfaces.
- L1–L5 read methods return empty collections (ready for M32.3 implementation).
- `IDimensionLabelRegistry` injected — infrastructure no longer hardcodes Customer dimension labels.
- `HasAnyCurrentMetrics` filters out `EA-META-*` and `EA-DIM-*` rows.

---

## KPI Metadata Improvements

All three customer-pack KPIs now include:

- `ValueType`, `AggregationType`, `TimeGrain`
- `DisplayPrecision`, `NullableBehavior`
- `EvidenceFilterDimension`, `SourceDomain`
- `ApplicableEntityTypes`, `DefinitionVersion`

`IKpiRegistry.ValidatePack(packId)` returns KPI IDs missing from registry.
