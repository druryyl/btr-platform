# M32.B1 — Entity Analytics Historical Backfill Implementation Plan

**Status:** Architect deliverable — ready for Implementer handoff

**Date:** 2026-06-25  
**Architecture:** [m32.b1-historical-backfill-architecture.md](./m32.b1-historical-backfill-architecture.md)

## Objective

Implement a one-time, production-safe historical replay pipeline that seeds **36 months** of Entity Analytics snapshots (L1, L2, L5 minimum) by reusing existing aggregators, producers, and L0–L5 engines.

**Out of scope:** Recurring scheduled backfill; profile UI changes; cloud sync changes.

## Risk Assessment

### Technical risks

| Risk | Likelihood | Impact | Mitigation |
| ---- | ---------- | ------ | ---------- |
| Live `SaveMonthlyHistory` MERGE cannot overwrite closed months | Certain | High | `ReplaceMonthlyHistoryForPeriod` backfill method (ADR-004) |
| DAL lacks AsOf piutang/stok | Medium | High | Add DAL overloads in B1.3; document approximation limits |
| L2 rank non-determinism on tie-break | Low | Medium | Sort by EntityCode before rank assignment |
| Item 36-mo exceeds maintenance window | High | Medium | Active subset; staged rollout; B1.9 optimizations |
| RepHistory KPI mapping drift | Low | Medium | Reconciliation tests in B1.5 |

### Operational risks

| Risk | Likelihood | Impact | Mitigation |
| ---- | ---------- | ------ | ---------- |
| Operator runs backfill without `--confirm` | Low | Low | Worker rejects writes without confirmation |
| Live refresh during backfill corrupts ranks | Medium | High | Entity-type mutex (architecture §10) |
| Resume confusion after partial failure | Medium | Medium | Checkpoint UI in logs; runbook in B1.8 |
| Unauthorized execution | Low | High | Admin-only; audit job table |

### Performance risks

| Risk | Likelihood | Impact | Mitigation |
| ---- | ---------- | ------ | ---------- |
| SQL Server CPU saturation | High (Item) | High | Off-peak; `max-parallel=1`; Item last |
| Tempdb growth from large sorts | Medium | Medium | Batch size tuning; monitor |
| Long transaction locks | Medium | Medium | One transaction per period, not per entity |

### Business assumptions

| Assumption | Risk if wrong | Mitigation |
| ---------- | ------------- | ---------- |
| Item → Supplier immutable for 36 mo | Wrong principal attribution in historical Supplier relationships | ADR-002; PO confirmed |
| Transactional tables complete for 36 mo | Gaps in early months | Pre-flight row count report in dry-run |
| Current aggregator rules acceptable for history | Semantic drift vs past business rules | Document in ADR-002; label backfilled data in runbook |
| Piutang AsOf reconstruction acceptable | Trend inaccuracy for open balance | Phase piutang KPIs after omzet; PO sign-off |
| 36-month retention sufficient | Profile gaps beyond window | `HistoryRetentionMonths` config |

## Production Runbook (Summary)

1. Notify stakeholders; schedule maintenance window for Item.
2. Verify live workers paused or mutex tested for target entity type.
3. Run dry-run: `--dry-run --entity-type All --from-period … --to-period …`
4. Review checkpoint row counts and storage estimate.
5. Execute: `--confirm BACKFILL --entity-type Salesman` → Customer → Supplier → Item.
6. On failure: inspect `LastError` on checkpoint; fix; `--force --from-period YYYY-MM`.
7. Validate profiles (sample entities per type).
8. Resume live workers; verify forward month accumulation.
